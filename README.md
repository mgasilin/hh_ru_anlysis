# Анализ спроса и предложения на рынке труда разработки и машинного обучения

Групповой проект №2 по дисциплине "Современные методы анализа данных и машинного обучения"

Команда: 46

Участники: Гасилин Максим, Солдатова Софья, Шалашов Дмитрий, Юшинский Андрей

## О проекте

В этом проекте мы решили посмотреть на рынок труда разработки и машинного обучения с двух сторон:

С одной стороны у нас есть вакансии с hh.ru, и это спрос со стороны работодателей

С другой стороны - пользователи GitHub и их репозитории, и это уже предложение со стороны потенциальных кандидатов

Наша идея в том, чтобы сопоставить вакансии и пользователей GitHub и понять, какие навыки сейчас реально нужны рынку, насколько они встречаются в открытых проектах и кто из разработчиков ближе всего к ожиданиям работодателей

## Бизнесовая часть

### Цель

Наша основная цель - собрать единый датасет по связке "вакансия - пользователь GitHub" и на его основе узнать, какие навыки и технологии чаще всего встречаются в вакансиях, насколько эти требования закрываются реальными проектами на GitHub и каких пользователей можно считать наиболее близкими к запросам работодателей

### Источники данных

В проекте мы используем два основных источника данных:

1. hh.ru

С hh.ru мы парсим технические вакансии, связанные с разработкой и машинным обучением, и берем из них названия вакансий, зарплаты, навыки, компании и тд. Суммарно их вышло более 15 тыс.


2. GitHub

Через GitHub API мы собираем пользователей и их репозитории

В выборку попадают пользователи GitHub из России, после чего мы оцениваем их по содержимому репозиториев, используемым языкам и технологиям

Дополнительно по репозиториям мы берем описание, темы, README и распределение языков

Итого вышло 5 запросов:

* `/search/user`
* `/users/id`
* `/users/id/repos`
* `repos/id/readme`
* `/repos/id/languages`

### Что получаем на выходе

На выходе у нас получается общий датасет, в котором для каждой вакансии можно найти близких пользователей GitHub по стеку, языкам, тематикам и текстовым признакам их репозиториев

На основе этого датасета мы считаем метрики соответствия вакансий стеку пользователей GitHub, анализируем востребованность технологий и строим графики по спросу и предложению

### Практическая ценность

Проект может принести практическую ценность в виде:

- анализа спроса на навыки и технологии
- оценки того, какие пользователи и какие технологии ближе всего к ожиданиям работодателей
- рекомендаций разработчику по тому, чего ему не хватает в портфолио


## Техническая часть

### Как устроены данные

В проекте есть два основных блока данных:

- вакансии hh.ru
- пользователи GitHub
- репозитории GitHub

Связка внутри GitHub-части идет через логин пользователя

### Данные hh.ru

#### Как собирали

Сбор был в два шага

Сначала `parsing/parser.py` проходит по поисковым запросам и сохраняет результаты выдачи в `new_links.json`

Потом `parsing/get_details.py` открывает карточки вакансий и добирает содержательные поля

#### Что хранится в первом файле

В `new_links.json` есть служебные поля `last_query_index` и `last_page`. Они нужны только для возобновления парсинга

Основной массив лежит в `vacancies`. Каждая запись - это одна вакансия из поисковой выдачи

Для вакансии сохраняются:

- `id` - технический идентификатор записи
- `query` - поисковый запрос
- `title` - название вакансии
- `link` - ссылка на карточку
- `page` - страница выдачи

Это еще не полная карточка вакансии. На этом этапе у нас только данные поиска

#### Что хранится после добора деталей

После второго шага для вакансии появляются основные содержательные поля:

- `company`
- `salary`
- `description`
- `skills`

Также сохраняются `id`, `query`, `title` и `link`

#### Типы данных и пропуски

По типам здесь все довольно просто:

- `title`, `company`, `salary`, `description`, `query`, `link` - строки
- `skills` - список строк

Главная особенность hh-части в том, что пропуски не сводятся к `NaN`

Здесь типичны такие филлеры:

- пустая строка для `company` или `description`
- строка `Не указана` для `salary`
- пустой список для `skills`

Если пропуски в компании или ЗП просто опускались - они нужны для вывода в UI и не критичны, вакансии с пропусками в навыках и описании просто не рассматривались

#### Что важно для анализа

Для дальнейшего мэтчинга и анализа нам в первую очередь нужны:

- `description`
- `skills`

`skills` - самый структурированный блок вакансии

`description` нужен для более гибкого извлечения технологий и обязанностей

`salary` хранится как текст, поэтому для численного анализа его нужно разбирать отдельно - для этого на одном из этапом из него сначала вычленялись блоки цифр, затем они по расположению объединялист в числа и потом уже преобразовывались в нужный тип данных

### Данные GitHub API

В GitHub-части используются две итоговые таблицы:

- `users_filtered.csv`
- `repos_filtered.csv`

Размеры таблиц такие:

- `users_filtered.csv` - 1000 строк и 20 столбцов
- `repos_filtered.csv` - 69941 строка и 30 столбцов

При этом один столбец в каждом файле - это технический индекс после сохранения CSV Unnamed: 0 - в анализе он не нужен


Поэтому содержательных признаков в `users_filtered.csv` - 19, а в `repos_filtered.csv` - 29

Полных дублей и дублей по `id` в этих таблицах нет

### users_filtered.csv

#### Что это за объект

Одна строка - один GitHub аккаунт

Ключевое поле - `login`

С ним дальше связываются репозитории через `sample_user_login`

#### Какие поля здесь важны

В таблице есть несколько групп признаков

Идентификация и ссылки:

- `login`, `id`, `html_url`, `type`

Поля профиля:

- `name`, `bio`, `location`, `company`, `blog`, `twitter_username`, `hireable`

Счетчики:

- `followers`, `following`, `public_repos`, `public_gists`

Даты:

- `created_at`, `updated_at`


#### Типы данных

В таблице есть:

- строковые поля
- целочисленные счетчики
- булево поле `hireable`
- даты

Поле `type` принимает два значения:

- `User` - 973 объекта
- `Organization` - 27 объектов

#### Пропуски

Пропуски здесь в основном связаны с тем, что пользователи сами не заполняют профиль

Самые заметные пропуски:

- `twitter_username` - 842
- `hireable` - 690
- `company` - 583
- `blog` - 460
- `bio` - 453
- `name` - 51

У `hireable` есть важная особенность: среди непустых значений встречается только `True`, остальные записи пустые

#### Что важно учитывать

Распределения счетчиков сильно скошены вправо

То есть у большинства пользователей умеренные значения, а большие аккаунты образуют длинный хвост

### repos_filtered.csv

#### Что это за объект

Одна строка - один репозиторий

Это основной источник информации о стеке, активности и тематике проектов

Ключ связи с пользователем - `sample_user_login`

#### Какие поля здесь важны

Для проекта наиболее полезны следующие группы признаков

Основная идентификация:

- `id`, `name`, `full_name`, `sample_user_login`, `owner_login`, `owner_type`

Текстовые признаки:

- `description`, `readme`, `topics`

Языки и стек:

- `language`, `lang_bytes`, `lang_pct`

Активность и популярность:

- `stargazers_count`, `forks_count`, `open_issues_count`, `size`

Статусы и служебные поля:

- `fork`, `private`, `archived`, `disabled`, `default_branch`, `visibility`
- `license_spdx_id`, `license_name`, `watchers_count`
- `created_at`, `updated_at`, `pushed_at`

Как и в таблице пользователей, `Unnamed: 0` - это лишний технический индекс

#### Типы данных

В таблице есть:

- строки
- числовые счетчики
- булевы признаки
- даты
- строковые представления списков и словарей

Здесь важно помнить, что:

- `topics` хранится как список в строковом виде
- `lang_bytes` хранит словарь языков и объемов
- `lang_pct` хранит словарь языков и долей

#### Базовые характеристики

По `owner_type` распределение такое:

- `User` - 68288 репозиториев
- `Organization` - 1653 репозитория

По статусам:

- форков - 38801
- нефорков - 31140
- архивных - 2621
- отключенных - 133
- приватных - 0

#### Пропуски и пустые структуры

У этой таблицы главная особенность в том, что отсутствие данных кодируется по-разному

Обычные `NaN` чаще всего встречаются в:

- `license_spdx_id` - 32786
- `license_name` - 32786
- `language` - 24353
- `description` - 14843
- `readme` - 10126

Но кроме `NaN` нужно учитывать и пустые структуры:

- `topics = []` у 63491 репозитория
- `lang_bytes = {}` у 7805 репозиториев
- `lang_pct = {}` у 7808 репозиториев
- пустой `readme` строкой встречается еще в 29 случаях

То есть для GitHub-части отсутствие информации - это не только `NaN`, но и `[]`, `{}` или пустой текст

#### Что важно для анализа

Нужно различать три языковых поля

- `language` - только основной язык репозитория
- `lang_bytes` - полный состав языков в байтах
- `lang_pct` - доли языков в проекте

Для реального описания стека `lang_bytes` и `lang_pct` намного полезнее, чем одно поле `language`

Еще одна техническая особенность - `watchers_count` в этой выгрузке полностью совпадает со `stargazers_count`, поэтому как отдельный признак почти не добавляет новой информации

Как и у пользователей, числовые счетчики здесь сильно скошены: у большинства репозиториев мало звезд, форков и issues, а заметные значения дает небольшой хвост популярных проектов

### Производные данные

На базе `repos_filtered.csv` в EDA строится длинная таблица языков

Для этого раскрываются `lang_bytes` и `lang_pct`, после чего получаем наблюдения вида:

- пользователь
- репозиторий
- язык
- объем языка
- доля языка

После фильтрации очень маленьких значений по порогу 5 перцентиля по `lang_bytes` получается таблица размером 183811 x 5

Эта форма данных удобнее для анализа технологий, потому что позволяет смотреть не только на главный язык, но и на весь состав проекта

### Что в итоге используем в мэтчинге

Для вакансий ключевыми остаются:

- `description`
- `skills`

Для репозиториев:

- `language`
- `lang_bytes`
- `lang_pct`

Для пользователя как кандидата важен не один отдельный репозиторий, а совокупный профиль по всем его проектам

#### Мэтчинг

С HuggingFace была взять [модель](https://huggingface.co/Nashhz/SBERT_KFOLD_Job_Descriptions_to_Skills) для семантического сравнения текста. В нее передавались сначала описание вакансии и навыки пользователя, затем навыки вакансии и навыки пользователя. Две полученные оценки от 0 до 1 брались с весом 0.5, формируя итоговую оценку

### Коротко о качестве данных

Данные в проекте достаточно прозрачные и хорошо связаны между собой

Главная сложность не в большом количестве мусора, а в том, что часть пропусков спрятана в текстовых филлерах и пустых структурах

Для hh.ru данные более текстовые и полуструктурированные

Для GitHub данные богаче и лучше структурированы, но для честного описания навыков нужно смотреть не только на основной язык, а на весь языковой состав, README, topics и текстовые описания
